# Repository Guidelines

## Project Structure & Module Organization
CalmsToolkit is a Go module that ships three CLI binaries from the root: `media-requests.go`, `media-streams.go`, and `media-calendar.go`. Supporting assets live alongside them; research notes are under `docs/`, example flows in `EXAMPLES_MEDIA_REQUESTS.md`, and reusable env defaults in `.env.example`. The `bin/` directory is generated by builds, while the Makefile drives reproducible compilation.

## Build, Test, and Development Commands
Use the Make targets to stay consistent: `make build` compiles the three binaries for your host, `make build-all` cross-compiles for common OS/arch pairs, `make test` runs `go test ./...`, and `make tidy` refreshes module metadata. During iteration you can run a tool directly, e.g. `go run -tags mediarequests media-requests.go -url ...`, but commit the compiled output only to `bin/`.

## Coding Style & Naming Conventions
Format every change with `gofmt` (or `go fmt ./...`) before committing; the project mirrors idiomatic Go spacing and grouping. Keep exported types, funcs, and consts in UpperCamelCase with doc comments, use lowerCamelCase for locals, and reserve ALL_CAPS for environment variable labels surfaced to users. Group related constants and config structs to keep CLI flag wiring readable.

## Testing Guidelines
Place table-driven tests in `_test.go` files beside the feature under test and cover both happy paths and API failure modes (mock HTTP responses where feasible). Target at least the functionality added in each PR; run `make test` or `go test ./...` before pushing, and capture notable coverage regressions in the PR notes.

## Commit & Pull Request Guidelines
Match the existing history by writing imperative, present-tense commit subjects (`Add media calendar filters`). Squash incidental fixups locally so each commit is reviewable. Pull requests should link the relevant issue, summarize behavioral changes, call out new flags or env vars, and include screenshots or sample CLI output when they clarify UX changes. Flag any security-sensitive configuration updates explicitly.

## Configuration & Secrets
Copy `.env.example` into a private `.env` when experimenting; never check real Overseerr, Plex, or Jellyfin tokens into Git. Local overrides should rely on shell exports or direnv, and scrub logs or fixtures before sharing them in reviews.
